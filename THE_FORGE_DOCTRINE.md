# The Forge Doctrine

## Core Principle

**Build nothing that cannot be repaired.**

## The Five Laws

### 1. Transparency Over Convenience
If it cannot be examined, it cannot be trusted.

- BootForge: Typed confirmation, evidence manifests
- Phoenix Key: Visible authority, audit trails
- Engine: Explicit Vulkan barriers, profiled frames
- Reforge OS: No hidden layers

### 2. Determinism Over Magic
If the same input doesn't produce the same output, it's wrong.

- BootForge: Hash-verified images, reproducible builds
- Engine: Fixed timestep physics, deterministic jobs
- All tools: Version-locked, hash-addressed

### 3. Offline Sovereignty
The cloud is optional, not required.

- BootForge: USB-resident, no network
- Phoenix Key: Local authentication
- Engine: Offline asset compilation
- All systems: Self-contained

### 4. Evidence Over Assumption
Nothing happens without a receipt.

- BootForge: Manifests, signatures, reports
- Phoenix Key: Audit chains, capability history
- Engine: Profiling, frame dumps, state snapshots
- All systems: Logs, hashes, attestations

### 5. Repair Over Replacement
If it breaks, you can fix it. If you can't fix it, it doesn't belong to you.

- BootForge: Repairable disk images
- Phoenix Key: Revocable authority
- Engine: Extensible, not monolithic
- All projects: Open to inspection

## Commercial Truth

You don't sell products.
You sell a worldview:

**"If it matters, you should see it happen."**

This worldview:
- Scales from games → OS → infrastructure
- Makes sense to shops, auditors, creators
- Builds trust without marketing tricks

## Filter for Future Projects

If a project:
- ❌ Hides complexity
- ❌ Cannot be examined
- ❌ Cannot be repaired
- ❌ Pretends to be "final"

**You don't build it.**

That's how you avoid burnout and junk projects.
