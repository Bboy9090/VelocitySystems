# Phoenix–Velocity Governance Charter (v1.0)

## Purpose

To ensure the Universal Core is operated with integrity, restraint, and continuity—regardless of product doctrine.

## Core Truths (Immutable)

1. **The Universal Core always records reality**
   - No doctrine may disable core logging
   - No doctrine may alter audit chains
   - No doctrine may suppress evidence

2. **Doctrine never alters history**
   - Past events are immutable
   - History cannot be rewritten
   - Only interpretation changes

3. **No employee may suppress or falsify audit data**
   - Zero tolerance for data manipulation
   - All modifications must be logged
   - Violations trigger immediate suspension

4. **Short-term sales pressure never overrides long-term truth**
   - Revenue does not justify lies
   - Customer requests cannot disable core
   - Marketing claims must match reality

5. **Survivability > velocity when conflict arises**
   - If core integrity is at risk, slow down
   - If evidence is compromised, stop
   - If truth is questioned, reveal

## Separation of Powers

### Doctrine Owners
- **Control**: Defaults, UI, messaging, product experience
- **Cannot**: Alter core enforcement, disable logging, modify audit chains
- **Responsibility**: Product success, user experience, market fit

### Core Stewards
- **Control**: Audit system, policy engine, evidence export, truth escrow
- **Cannot**: Influence product marketing, change doctrine defaults, hide features
- **Responsibility**: System integrity, legal defensibility, audit readiness

### System Owners
- **Control**: Dual-key authority, kill-switch, truth escrow unsealing
- **Cannot**: Act alone (requires dual-key), bypass audit, suppress evidence
- **Responsibility**: Ultimate system integrity, emergency response, legal compliance

## Internal Violations (Zero Tolerance)

### Category 1: Core Integrity Violations
- Disabling core logging
- Modifying audit chains
- Backdating records
- Suppressing evidence

**Consequence**: Immediate suspension, internal investigation, permanent audit marker

### Category 2: Doctrine Violations
- Marketing claims that contradict doctrine reality
- UI that hides required approvals
- Defaults that bypass core enforcement
- Messaging that promises what doctrine cannot deliver

**Consequence**: Product freeze, marketing review, doctrine correction

### Category 3: Truth Escrow Violations
- Attempting to access escrow without authorization
- Attempting to delete escrow data
- Attempting to modify escrow records

**Consequence**: Immediate termination, legal review, evidence preservation

## Audit Requirements

### Internal Audits
- Monthly integrity verification
- Quarterly doctrine compliance review
- Annual truth escrow verification
- Ad-hoc on suspicion

### External Audits
- Customer audits: Full evidence export
- Regulatory audits: Truth escrow unsealing (if authorized)
- Legal proceedings: Complete audit chain export

## Escalation Path

1. **Suspected Violation**: Report to Core Steward
2. **Investigation**: Core Steward + System Owner (one key)
3. **Resolution**: System Owners (dual-key) if severe
4. **Documentation**: All violations logged in audit chain

## Amendment Process

This charter can only be amended by:
- Unanimous System Owner approval (dual-key)
- Core Steward review
- Doctrine Owner notification
- Audit trail of amendment

## Acknowledgment

All employees with access to core systems must acknowledge this charter annually.
